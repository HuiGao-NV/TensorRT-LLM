# Change Analyzer

Analyze the git changes in the current branch relative to its remote tracking branch, and write a structured `review.md` in the current working directory.

## Workflow

### Step 1: Discover Changed Files

Get the list of changed files compared to upstream:

```bash
git diff --name-status @{upstream}..HEAD
```

If `@{upstream}` fails (no upstream configured), fall back:

```bash
git diff --name-status origin/main..HEAD 2>/dev/null || \
git diff --name-status origin/master..HEAD 2>/dev/null
```

If everything fails, warn the user:
> "No upstream branch is configured. Run `git branch --set-upstream-to=origin/<branch>` to set one. Falling back to origin/main."

Also note any uncommitted changes with `git status --short` — mention them in the report but don't include them in the diff analysis (focus on committed changes).

### Step 2: Gather Commit History

Get all commits from upstream to HEAD, excluding merge commits:

```bash
git log @{upstream}..HEAD --format="%h %s%n%b" --no-merges
```

Capture both the subject line and body for each commit — commit bodies often contain important context about *why* a change was made.

If the branch is already up to date with upstream, say so and stop — there's nothing to analyze.

### Step 3: Get the Diff

Retrieve the full diff:

```bash
git diff @{upstream}..HEAD
```

If the output is very large (over ~1000 lines), prioritize the files with the most changes. You don't need to read every line — your goal is to understand what the code is doing, not to audit every detail.

### Step 4: Analyze

Using the commit messages and diff together, analyze three dimensions:

**1. Purpose — What is this change for?**
Synthesize commit messages and code changes into a clear statement of intent. What problem does this solve, or what feature does it add? Commit messages tell you the *what*; the diff tells you the *how* — combine them for the *why*.

**2. Module — What part of the codebase is affected?**
Identify the modules, packages, or subsystems touched. Group related files. Note if changes span multiple concerns (e.g., core logic + tests + config).

**3. Impact — What are the effects?**
Be specific. "Improves performance" is weak. "Reduces memory allocation in the inference loop by reusing tensor buffers" is strong. Consider:
- **Functionality**: Behavioral changes, new capabilities, bug fixes
- **Performance**: Speed, memory, or resource usage changes
- **Usage**: API, interface, config, or CLI changes that callers need to know about

### Step 5: Write review.md

Write the analysis to `review.md` in the current working directory.

## Output Format

```markdown
# Code Change Review

**Branch:** `<current-branch-name>`
**Base:** `<upstream-ref>`
**Commits:** <N> commits
**Files changed:** <N> files

---

## Summary

<1–3 sentence plain-English summary of what this change does overall.>

---

## Changed Files

| File | Change |
|------|--------|
| `path/to/file.py` | Modified — brief description |
| `path/to/new_file.py` | Added — brief description |
| `path/to/deleted.py` | Deleted |

---

## Commit History

<Commits, newest first>

- `abc1234` feat: add streaming support for batch inference
- `def5678` fix: handle empty input edge case

---

## Analysis

### Purpose

<What this change is for — the problem solved or feature added>

### Module

<What part of the codebase is affected, with groupings if multiple subsystems>

### Impact

**Functionality**
<How behavior changes, or "No behavioral changes.">

**Performance**
<Performance implications, or "No significant performance impact identified.">

**Usage**
<API/interface/config changes callers should know about, or "No interface changes.">

---

*Generated by change-analyzer*
```

## Edge Cases

- **No upstream**: Warn the user and fall back to `origin/main` / `origin/master`. If neither exists, ask the user what to compare against.
- **Nothing to analyze**: If the branch is up to date with upstream, say so and skip writing the file.
- **Very large diffs**: Analyze the highest-impact files first. Note in the report if parts of the diff were summarized due to size.
- **Only uncommitted changes**: If there are no commits ahead of upstream but there are staged/unstaged changes, offer to analyze just those using `git diff HEAD` instead.
